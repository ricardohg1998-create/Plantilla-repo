# M4: UI Kit + Core Sections (Data-Driven)

## OBJETIVO (OBJ)
Construir un kit de componentes UI sólido y las secciones clave de cualquier web corporativa, totalmente conectadas a datos (JSON/TS) y no harcodeadas en el JSX.

## ALCANCE (ALC)
- Desarrollo de componentes atómicos en `shared/ui`.
- Implementación de secciones core (Header, Hero, Servicios, etc.).
- Gestión de formularios seguros.

## PASOS
1.  **UI Kit Base**:
    - Crear botones, inputs, cards, modales con variantes (primary, outline, ghost).
    - Asegurar estados: hover, focus-visible, active, disabled, loading.
2.  **Secciones Core**:
    - **Header**: Responsive (Mobile Drawer), sticky inteligente.
    - **HeroEditorial**: Soporte para vídeo/imagen, layout asimétrico.
    - **Servicios/Proceso**: Bento grid o timeline vertical.
    - **Casos de Éxito**: Card con imagen, filtros por categoría.
    - **Blog**: Listado paginado + Detalle de post (renderizado de MDX/CMS).
    - **Contacto**: Layout con mapa (opcional) y formulario.
3.  **Formularios "Seguros"**:
    - Usar `react-hook-form` + `zod` para validación.
    - Implementar Honeypot field (hidden input).
    - Rate limit en API route (si es Next.js).
    - Feedback claro: toast de éxito/error.

## CRITERIOS DE ACEPTACIÓN (AC)
- [ ] Todas las secciones leen su contenido de props o data files, NADA harcodeado.
- [ ] Formularios validados con Zod y resistentes a spam básico.
- [ ] Diseño responsive comprobado en móvil (375px) y desktop.

## EJECUCIÓN (EJ)
```tsx
// Ejemplo Hero data-driven
<HeroSection
  title={data.hero.title}
  cta={{ label: data.hero.cta, href: "#contact" }}
  image={data.hero.image}
/>
```

## RIESGOS
- Sobre-ingeniería de componentes (hacerlos demasiado flexibles y difíciles de mantener).
- Formularios que rebotan usuarios legítimos por validación demasiado estricta.
