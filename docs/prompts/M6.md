# M6: A11y + Motion (Inclusivo y Premium)

## OBJETIVO (OBJ)
Garantizar que la experiencia "premium" no excluya a nadie. La accesibilidad es sinónimo de calidad de ingeniería. Las animaciones deben aportar valor, no marear.

## ALCANCE (ALC)
- Auditoría básica de accesibilidad (Axe).
- Implementación de `prefers-reduced-motion`.
- Estrategia de Motion Design (Micro vs Macro).

## PASOS
1.  **Fundamentos A11y**:
    - Focus visible: NUNCA `outline: none` sin reemplazo claro.
    - Semántica: Uso estricto de `<main>`, `<nav>`, `<aside>`, `<h1>-<h6>`.
    - Formularios: Labels explícitos (no solo placeholders). Mensajes de error vinculados con `aria-describedby`.
    - Contraste: Validar colores automáticos (especialmente texto sobre imágenes).
2.  **Motion Design Responsable**:
    - Usar `framer-motion` (Next) o `motion` (Vite) con un wrapper global `MotionConfig`.
    - Respetar `reducedMotion: "user"`. Si el usuario pide reducir, las animaciones de scroll se desactivan o simplifican a fade-in.
    - Categorías:
        - **Macro**: Transiciones de página, Hero reveal (Lento, suave).
        - **Micro**: Hover botones, switch toggles (Rápido: <200ms).

## CRITERIOS DE ACEPTACIÓN (AC)
- [ ] Lighthouse Accessibility score > 95.
- [ ] Navegación completa por teclado funcional (Tab, Enter, Space).
- [ ] Al activar "Reducir movimiento" en el SO, la web no hace paralaje ni zooms agresivos.

## EJECUCIÓN (EJ)
```tsx
// Hook para movimiento seguro
const shouldReduce = useReducedMotion();
const variants = shouldReduce ? { visible: { opacity: 1 } } : { visible: { x: 0, opacity: 1 } };
```

## RIESGOS
- Diseños "Fancy" que confían solo en hover y no funcionan en táctil.
- Scroll-jacking (secuestrar el scroll) que rompe la navegación nativa.
